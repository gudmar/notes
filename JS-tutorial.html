npm

Node package manager:
  • Default package manager for Node.js
  • Repository for open source Node.js projects
  • Written with js
  • Name of company managing repo attached to npm tool
  • cli for installation of packages and manage their version
  • npm on package installation creates link to installed package:
    • Local: ./node_modules/.bin/
    • Global: usr/local/bin or %AppData%/npm for Windows
  • ./node_modules/.bin/some-package - a run command
  • package-name This should work if package is installed globaly
  • Only after creation a package.json package can be run with npm run package-name command
NPX: node package execute:
  • Is a application for executing node modules downloaded from npm repo
  • Is capable of installing locally and 'temporarly' packages before running them if they are not found
  • Simple command like npx some-package is capable of installing localy package and run it in one command. Just for tests.
  • Packages are run either localy without specifying their local path or from global path, or installed locally and run if they are not found anywhere.
  • This is more like running packages from external repo, then my own created js files.
  • Good for running just say npx create-react-app as this will be run only once per developement cycle of application to install environment
Yarn - like npm, but stores packages in cashe and makes it possible to install them from cash in case of problems with npm repo.
Node.js: Environment to develop server side apps in JS.

package.json

  • Used by npm to handle dependencies, store metadata, store commands needed to compile, run scripts
  • Node.js is aware of only 'name' and 'version' fields
  • created with npm init command. Then can be extended
  • When npm install some-package --save-dependencies is run then new data are added to package.json, so it can install those dependencies with project with one command later on
  • Should be stored in repo

package-lock.json vs npm-shrinkwrap.json

  • 2.3.4 - package version. 2 means a major feature, 3 is a minor feature, but still capable of causing huge problems with the application, 5 is a bug fix. Should not affect regression
  • ^2.3.4 - this ^ indicates, that a newest minor feature version should be installed, ~ maens that latest patch version needs to be installed.
  • package-lock.josn will ensure, that the exect same packages are installed when npm install is run
  • npm-shrinkwrap.json does the same as package-lock.json, however it is stronger (it coveres package-lock.json) and it can be put into a subfolder, where package-lock is ignored everywhere outside root folder of application.
  • npm-shrinkwrap can be published - available if I create a dependency package
  • Shoud be commited. node_modules is not commited, so space is saved and every future user has the same package versions.
  • Lock file is not published to NPM software registry - so if I create a package it becomes a dependency to other software, so package_lock would not be in root folder and it would be igronred anyway. So my dependencies would be merged into a whole project dependency list.
  • paclage-lock file has complete tree of exect dependencies. Package.json has no dependencies of my dependencies, so it is a shoalow version, moreover contains not exect version, but only a version with possibility to install newer version of the file.

Iterators

  • Generator is an object having a next() method, returning next object from the collection
  • next() returns object with 2 properties: value and done
  • value is a value of next object, where done is true if iteratro got to last element of collection, and false if there still is an object
  • After getting to the last collection element there is no possibility to rewind an iterator
  • If object has an object having next function returning {value: _, done: _} under [Symbol.iterator] then it is iterable, and for..of or spread operator can be used on it

Example

        function makeIterator(array) {
            var nextIndex = 0;
        
            return {
               next: function() {
                   return nextIndex < array.length ?
                       {value: array[nextIndex++], done: false} :
                       {done: true};
               }
            };
        }
Or with ES6 
        class MyIterator{
            constructor(max){
                this.currentValue = 0;
                this.maxValue = max
                this.step = 1
            }
            next() {
                let nextVal = this.currentValue + this.step;
                this.currentValue++;
                return {
                    value: this.currentValue,
                    done: nextVal >= this.maxValue?true:false
                }
            }
            [Symbol.iterator]() {
                return {next: () => this.next()}
            }
        }
        let myIter = new MyIterator(10)
        console.dir(myIter)
        for(let i of myIter){
            console.log(i)
        }
        console.log([...myIter]) // This will be empty, as itrator is already used

Mouse position on side
  • MouseEvent.pageX: read only property indicating mouse left position relative to begining of page. Takes into account also not visible page parts, scrolling etc. If page is scrolled 200px, mouse is on 100px of viewport from left, pageX returns 300. Returned value is a Float.
  • MouseEvent.pageY: the same as page X, but returns position relative to top of whole document.
  • MouseEvent.clientX: read only property indicating mouse left position relative to current viewport (beginning of screen). No matter how much document is scrolled, returned value will be from begining of screen browser viewport. If clicked on left edge of browser edge, will return 0. Difference between this and seceenX is, that screen is physical screen, and client is from edge of browser. Browser edge might be not alligned with physical screen.
  • MouseEvent.clientY: similar to client X, however returns position relative to sceen top.
  • MouseEvent.movementX; MouseEvent.movementY difference in mouse coordinance between position before mouse movement and after.
  • MouseEvent.xOffset, MouseEvent.yOffset mouse position relative to targets object padding box. Experimental
  • MouseEvent.screenX, MouseEvent.screenY: this are coordinance from begining of physical screen.
  • MouseEvent.relatedTarget holds a target related to mouse event. Eg if mouse was over a div element, that div element will be returned.
